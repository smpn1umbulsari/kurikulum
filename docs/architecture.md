# System Architecture: Guru Spenturi v2

Dokumen ini mendefinisikan arsitektur teknis aplikasi Guru Spenturi v2 yang dibangun dengan Next.js App Router, Tailwind CSS, shadcn/ui, Supabase (Auth, Database, Storage), dan Capacitor untuk aplikasi Android.

---

## 1. Topologi & Aliran Data (Data Flow)

```text
+------------------+     +------------------------+     +------------------+
|   Web Browser    |     | Next.js Server Actions |     |  Supabase Auth   |
|   (Client App)   | <-> |  (Server Middleware)   | <-> | & Cookie Session |
+------------------+     +------------------------+     +------------------+
        ^                            ^                            ^
        |                            |                            |
        +----------------------------+----------------------------+
                                     |
                                     v
                        +--------------------------+
                        |   Supabase PostgreSQL    |
                        | (RLS enforced per table) |
                        +--------------------------+
```

* **Client Component**: Menangani interaksi pengguna, rendering state, spreadsheet input UI, drag & drop, dan layout responsif.
* **Server Middleware**: Memotong setiap HTTP request untuk memeriksa validitas sesi cookie. Jika valid, meneruskan request. Jika tidak valid, mengalihkan ke `/login`. Jika mode pemeliharaan aktif dan peran pengguna bukan admin/superadmin, mengalihkan ke `/maintenance`.
* **Supabase Client**: Terbagi menjadi `client.ts` (untuk browser/client component) dan `server.ts` (untuk server components, server actions, dan middleware) menggunakan cookie session.
* **Row Level Security (RLS)**: Semua query langsung dari klien ke database diverifikasi oleh engine RLS PostgreSQL di Supabase menggunakan token sesi JWT pengguna (`auth.uid()`).

## 3. Keamanan & Otorisasi

### 3.1 Autentikasi Supabase Auth
Sesi pengguna dikelola menggunakan cookie `sb-access-token` dan `sb-refresh-token` dengan bendera `HttpOnly`, `Secure`, dan `SameSite=Lax`. Token JWT didekripsi di sisi server Next.js pada setiap request untuk memastikan token belum kadaluarsa.

### 3.2 Row Level Security (RLS) di Supabase
Setiap tabel wajib mengaktifkan RLS (`ALTER TABLE nama_tabel ENABLE ROW LEVEL SECURITY;`).
Kebijakan RLS (RLS Policies) ditulis dengan mencocokkan UUID pengguna dengan data peran di tabel `pengguna`. Contoh kebijakan RLS untuk tabel `nilai`:

```sql
-- Kebijakan SELECT untuk tabel nilai
CREATE POLICY select_nilai_policy ON nilai
  FOR SELECT
  USING (
    -- Admin & Urusan bisa melihat semua nilai
    (SELECT role FROM pengguna WHERE id = auth.uid()) IN ('admin', 'urusan')
    OR
    -- Guru hanya bisa melihat nilai untuk kelas Real & Mapel yang diajar
    (
      (SELECT role FROM pengguna WHERE id = auth.uid()) = 'guru'
      AND mata_pelajaran_id IN (
        SELECT mata_pelajaran_id FROM pembagian_mengajar_real pmr
        JOIN guru g ON pmr.guru_id = g.id
        WHERE g.pengguna_id = auth.uid() AND pmr.semester_id = nilai.semester_id
      )
    )
  );
```

### 3.3 Middleware Route Protection
File `src/middleware.ts` memproteksi semua route di bawah `(protected)/*` dengan memeriksa sesi cookie. Jika pengguna mencoba mengakses route admin (`/admin/*`) namun perannya adalah `guru`, middleware mengembalikan status 403 Forbidden atau me-redirect ke dashboard utama.

## 4. Mode Pemeliharaan (Maintenance Mode)

Mode pemeliharaan diatur secara terpusat pada tabel `pengaturan_sekolah` melalui kolom `mode_pemeliharaan` (boolean).
1. **Pemeriksaan Middleware**: Middleware Next.js memeriksa status `mode_pemeliharaan` dari cache database/Supabase pada setiap request.
2. **Kondisi Pengalihan**:
   * Jika `mode_pemeliharaan` bernilai `true` dan pengguna bukan admin/superadmin, request langsung dialihkan ke `/maintenance`.
   * Halaman `/maintenance` mengecualikan pemeriksaan agar tidak terjadi *infinite loop redirect*.
   * Admin yang sedang login akan melihat banner pemberitahuan kecil di bagian atas aplikasi bahwa sistem sedang dalam masa pemeliharaan, namun tetap dapat beroperasi normal.

## 5. Integrasi Capacitor (Aplikasi Android)

Untuk membungkus aplikasi Next.js menjadi APK Android menggunakan Capacitor, aturan berikut diterapkan:
1. **Next.js Static Export**: Konfigurasi `next.config.js` diatur untuk menggunakan static export (`output: 'export'`) agar menghasilkan folder statis `out/` yang berisi HTML, CSS, dan JS murni.
2. **Sinkronisasi Capacitor**: Folder build statis tersebut diarahkan sebagai direktori web Capacitor di file `capacitor.config.ts`:
   ```typescript
   import { CapacitorConfig } from '@capacitor/cli';
   const config: CapacitorConfig = {
     appId: 'com.guruspenturi.v2',
     appName: 'Guru Spenturi',
     webDir: 'out',
     bundledWebRuntime: false
   };
   export default config;
   ```
3. **Build Pipeline**:
   * Jalankan `npm run build` untuk menghasilkan folder `out/`.
   * Jalankan `npx cap sync` untuk menyalin aset `out/` ke folder `/android/app/src/main/assets/public/`.
   * Buka Android Studio dan lakukan kompilasi rilis APK.
4. **Offline Fallback**: Di mobile, penanganan koneksi internet offline menggunakan layout penjelas khusus jika koneksi ke Supabase terputus.
